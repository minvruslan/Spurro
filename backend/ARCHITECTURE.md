# Architecture Rules (Hono + Modular Monolith)

## 1. Stack

- **Framework**: Hono.
- **Database**: PostgreSQL + Drizzle ORM.
- **Auth**: Better-auth.
- **Validation**: Zod via `@hono/zod-validator`.
- **Types**: `@spurro/shared` — domain schemas and types shared with frontend.

## 3. Authentication & Authorization

Authentication and authorization are handled by the **better-auth** library.

### Auth Instance

Configured in `src/core/auth-server/index.ts` (`authServer`):

- Email/password authentication is disabled.
- **Plugins**:
  - `admin()` — adds the `role` / `banned` fields and admin capabilities.
  - `magicLink()` — adds passwordless sign-in.

### Auth Schema

Better-auth tables live in a dedicated `src/core/database/auth-schema.ts`, separate from the
domain schema (`schema.ts`). This file is kept separate so the better-auth CLI can be re-run without
overwriting the hand-written domain schema. Both files are listed in `drizzle.config.ts`, so Drizzle Kit
treats them as one combined schema and produces a single migration set (`schema.ts` references the `user`
table for FK relations).

### HTTP Handler

In `src/app.ts`, all auth endpoints are delegated to better-auth:

```ts
api.on(["POST", "GET"], "/auth/*", (c) => authServer.handler(c.req.raw))
```

### Route Guards

Middlewares in `src/core/middlewares/`:

- `requireAuth` — resolves the session, rejects anonymous requests (401), and sets `userId`.
- `requireAdmin` — same as above plus a `role === "admin"` check (403 otherwise).

Protected routers mount `requireAuth` before their sub-routes.

### Admin Bootstrap

`bootstrapAdmin` (`src/core/bootstraps/bootstrapAdmin.ts`) runs on startup and creates the
initial admin user from `ADMIN_EMAIL` / `ADMIN_NAME` env vars if it does not already exist.

## 4. Modules

All business logic lives inside `src/modules/`. Each module groups all logic for a single domain entity.
Within each module, logic is organized in layers: routes → services → repository.

- One directory per domain entity — `modules/configs/`, etc.
- `index.ts` is the public API of each module.
- Shared logic goes to `src/core/`.

## 5. Module Layers

Default data flow:

```
route → service → repository → database
```

### Routes

HTTP boundary — validates the request and delegates to services, unaware of business logic.

- Validate request input with `zValidator`.
- Call service functions.
- Return JSON response.

### Services

Business logic — knows about the domain and the repository layer, not about HTTP.

- Orchestrate repository calls.
- Wrap multi-step operations in a transactions.
- Return domain types from `@spurro/shared`.

### Repository

Data access — knows about the database, not about business logic.

- Must not contain business logic.
- Keep each function in its own file.
- Accept `DbOrTx` to work both inside and outside transactions.
- Use `InferSelectModel` / `InferInsertModel` for DB-level types.

## 6. Monorepo Shared Package (`@spurro/shared`)

- Domain schemas and types are defined in `@spurro/shared`, not in the backend tree.
- Input validation schemas — used in routes via `zValidator`.
- Domain types (`Config`, `ConfigLimit`, etc.) — used as service return types.

## 7. Database Schema

The domain schema is defined in `src/core/database/schema.ts` using Drizzle table builders.
Auth tables live in `auth-schema.ts` and are generated by the better-auth CLI (see Authorization).
Migrations are managed with Drizzle Kit.

DB types are always inferred from the schema — never written manually:

```ts
type NewAccessGrant = InferInsertModel<typeof accessGrant>
type AccessGrantRow = InferSelectModel<typeof accessGrant>
```

## 8. Naming Conventions

- Route files: `camelCase` + `Route` suffix — `getConfigsRoute.ts`.
- Service files: `camelCase` + `Service` suffix — `getConfigsService.ts`.
- Repository files: `camelCase` verb phrase — `findConfigs.ts`, `countConfigsByProtocolType.ts`.
- Util files: `camelCase` describing what it returns — `createConfigFromDatabaseData.ts`.
- One function per file in `routes/`, `services/`, `repository/`.

## 9. Path Alias

`@/*` resolves to `src/*` (configured in `tsconfig.json`).

Note: `.js` extensions are required in imports (ESM output).
